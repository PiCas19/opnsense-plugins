# Security Configuration

The OPNsense Monitoring Bridge is designed to operate securely within a DMZ environment, implementing multiple layers of security to protect the application and the OPNsense firewall. This document outlines the security measures and best practices for configuring and maintaining a secure deployment.

## Authentication
The bridge uses JSON Web Tokens (JWT) for authenticating requests to protected endpoints (e.g., `/monitoring/system-stats`, `/emergency/block-ip`). Key configurations:
- **JWT_SECRET_KEY**: Set a strong, unique secret in `.env` (see `.env.example`). Example:
  ```plaintext
  JWT_SECRET_KEY=your-super-secure-jwt-secret-key-change-in-production
  ```
- **Token Generation**: Scripts like `scripts/emergency-block.sh` and `scripts/health-check.sh` generate JWT tokens with a limited expiration (1 hour) and specific roles (`admin`, `monitor`).
- **Protected Endpoints**: Require a valid JWT in the `Authorization: Bearer <token>` header.

To generate a test JWT token:
```bash
python3 -c "import os; from jose import jwt; from datetime import datetime, timedelta; payload = {'exp': datetime.utcnow() + timedelta(hours=1), 'iat': datetime.utcnow(), 'sub': 'test_user', 'role': 'admin'}; print(jwt.encode(payload, os.getenv('JWT_SECRET_KEY'), algorithm='HS256'))"
```

## IP Whitelisting
The bridge restricts access to specific IP ranges to prevent unauthorized access:
- **ALLOWED_IPS**: Comma-separated list of allowed IP addresses or ranges in `.env`. Example:
  ```plaintext
  ALLOWED_IPS=172.16.216.0/24,192.168.216.0/24,127.0.0.1
  ```
- **Middleware**: The FastAPI middleware (`app/middleware/authentication.py`) validates incoming request IPs against `ALLOWED_IPS`.
- **DMZ Restrictions**: Ensure `DMZ_NETWORK` (e.g., `172.16.216.0/24`) is correctly configured to limit access to the bridge.

## SSL/TLS Encryption
All API communications are secured with SSL/TLS:
- **Certificates**: Self-signed certificates are generated by `scripts/setup-bridge.sh` if not provided. Paths are set in `.env`:
  ```plaintext
  SSL_CERT_PATH=/app/certs/server.crt
  SSL_KEY_PATH=/app/certs/server.key
  SSL_CA_PATH=/app/certs/ca.crt
  ```
- **SSL Verification**: Set `OPNSENSE_VERIFY_SSL=false` and `SSL_VERIFY_PEER=false` for self-signed certificates. For production, use CA-issued certificates and set `SSL_VERIFY_PEER=true`.
- **TLS Version**: Enforces `SSL_MIN_VERSION=TLSv1.2` for secure connections.

To generate certificates:
```bash
./scripts/setup-bridge.sh
```

## DMZ Configuration
The bridge is designed for DMZ deployment:
- **Network Configuration**:
  - `DMZ_NETWORK=172.16.216.0/24`: Defines the DMZ subnet.
  - `DMZ_INTERNAL_NETWORKS=192.168.216.0/24`: Specifies accessible internal networks.
  - `DMZ_OUTBOUND_ALLOWED=smtp.gmail.com:587`: Limits outbound connections.
- **Offline Mode**: Enable `DMZ_OFFLINE_MODE=true` to buffer alerts during connectivity issues (`DMZ_ALERT_BUFFER_SIZE=1000`).
- **Batch Notifications**: Use `DMZ_BATCH_NOTIFICATIONS=true` to optimize notification delivery in DMZ (`DMZ_BATCH_SIZE=10`, `DMZ_BATCH_TIMEOUT=30`).

## Logging and Auditing
- **Log Files**: Logs are stored in `logs/` (e.g., `logs/bridge.log`, `logs/emergency.log`) with rotation settings:
  ```plaintext
  LOG_MAX_SIZE=10MB
  LOG_BACKUP_COUNT=5
  EMERGENCY_LOG_BACKUP_COUNT=10
  AUDIT_LOG_BACKUP_COUNT=20
  ```
- **Audit Logs**: Emergency actions (e.g., IP blocking) are logged to `logs/emergency.log` and backed up to `ALERT_BACKUP_FILE=/app/logs/alerts_backup.jsonl`.
- **Log Level**: Set `LOG_LEVEL=INFO` in `.env` (or `DEBUG` for development).

Check logs:
```bash
tail -f logs/bridge.log
```

## Best Practices
- **Secure `.env`**: Never commit `.env` to version control (ensured by `.gitignore`). Use `.env.example` as a template.
- **Rotate Secrets**: Regularly update `JWT_SECRET_KEY`, `OPNSENSE_API_KEY`, and `OPNSENSE_API_SECRET`.
- **Monitor Access**: Review `logs/audit.log` for unauthorized access attempts.
- **Production Certificates**: Replace self-signed certificates with CA-issued ones for production.
- **Backup Configuration**: Enable `BACKUP_ENABLED=true` and set `BACKUP_DIRECTORY=/app/backups` for regular configuration backups.
- **Test Security**: Run `scripts/test-api.sh` and `scripts/health-check.sh` to verify authentication and endpoint security.

For further configuration details, see [Configuration Guide](deployment/configuration.md).
